# Dotfiles Architecture Decisions (`dots/.memory/decisions.md`)

Active, canonical decisions for the `dots/` subtree. High-signal — read before
implementation. Scoped layer beneath repo-root
[`decisions.md`](file:///home/lowcache/.nix-config/.memory/decisions.md).

## D1 — Scoped memory lives at `dots/.memory`, never inside an app dir
Because each `dots/<app>/` is a whole-directory `mkOutOfStoreSymlink` into `~/.config`,
any file placed inside an app dir leaks into the live config (and into quickshell's QML
loader). `dots/` itself is not a symlink target, so the scoped `.memory/` and `.model/`
go there. Per-app notes use subdirs under `dots/.memory/`, not under the app dir.

## D2 — Independent history via `git subtree`, not submodule
The `dots/` tree must be trackable on its own history yet remain in the single nix-config
repo with no separate working repo and no submodule. `git subtree split/push/pull` gives a
derived, regenerable projection branch (`dots-history`) that can be published to a standalone
`dotfiles` remote and merged back, without moving any files — so `mkOutOfStoreSymlink` paths
are never disturbed. Submodules were rejected: they create a separate `.git`.

## D3 — Edit the palette, not generated theme outputs
The colorscheme is generated by `apply_theme.py` from a palette JSON
(`themes/amalgamation.json`). Generated files (`Appearance.qml`, `colors.conf`,
kitty/starship configs) are overwritten on every apply. Therefore color changes are made in
the palette JSON and re-applied; hand-edits to generated files are non-durable.

## D4 — Preserve hot-reload; no static Home-Manager dotfile outputs
Dotfiles are out-of-store symlinks specifically so edits are live without a rebuild. Do not
replace them with Home-Manager `text`/`source` outputs that would write static copies to
`~/.config`. Keep the out-of-store link to `dots/`.
